# v1.0.0 — Baseline (live release)

The reference point for all future versions. Documents *what exists* and *why it works the way it does*, so later designs stay consistent with it.

## Feature summary (as published)

- **11 ore generators**: Cobblestone, Coal, Copper, Iron, Redstone, Lapis Lazuli, Gold, Quartz, Emerald, Diamond, Netherite. (Code also contains an `oak_log` generator — treated as a soft-launched extra, see [generator_definitions.ts](../../scripts/definitions/generator_definitions.ts).)
- **Offline generation** on real-world time.
- **Manual withdrawal**: interact = 1 item, sneak+interact = up to 64 (1 stack).
- **Animated visual entity** per generator.
- **Storage caps** per tier (64–1024 in code).
- **Localization**: English + Spanish.
- Requires Bedrock **1.21.130+**, no experimental toggles.

## Architecture as-is

### One component for all generators
A single custom block component `idleoregen:generator` ([scripts/components/generator.ts](../../scripts/components/generator.ts)) handles `beforeOnPlayerPlace`, `onPlace`, `onPlayerInteract`, `onPlayerBreak` for every generator type. The block's `typeId` maps to a definition via `getGeneratorTypeFromBlockId`. Adding a generator type is data (a `GENERATORS` entry) + parallel content files, not new component code.

### Lazy, timestamp-based production (the core invariant)
There is **no per-tick production loop**. Each placed generator stores `lastInteraction` (a real-world `Date.now()` ms stamp). Produced amount is derived on demand:

```
elapsed = now - lastInteraction
cycles  = floor(elapsed / (interval * 1000))
stored  = min(storedAmount + cycles, cap)
```

On withdrawal, `lastInteraction` is advanced by **whole consumed cycles only** (`lastInteraction += cycles * intervalMs`), never to raw `now` — this preserves sub-cycle progress. This is exactly what makes offline generation "free": elapsed real time is the only input.

> **Invariant for all future work:** production must remain a pure function of elapsed real time and per-instance state. Do not introduce mechanics that require the chunk to tick in order for base production to occur.

### Persistence — World Dynamic Properties
Two JSON blobs, keys centralized in [storage_keys.ts](../../scripts/storage/storage_keys.ts):

- **Catalog** (`sb:w:c:generators`): the `GENERATORS` map serialized on world load (`initDatabaseAutoUpdate`). Single source of truth for interval, cap, output item, entity id, HUD glyph.
- **Placed state** (`sb:w:d:placed_state`): a **single** `Record<posKey, PlacedInstance>` for *every* placed generator, keyed `"dimensionId:x,y,z"`. CRUD in [instances/placed.ts](../../scripts/instances/placed.ts).

`PlacedInstance` already carries `ownerId` (unused by gameplay yet) and `placedAt`. `DB_VERSION` exists for migrating the placed data shape.

### Invisible block + visual entity
Generators render as a non-collidable entity, not the block. The block has a `sb:runtime` state (`0` = visible geometry, `1` = `empty_geometry`). On place: set `sb:runtime=1`, spawn the definition's `entityId` at block center, tag it `io:gen:<posKey>`. On break: find the tagged entity by proximity and remove it.

### HUD
[scripts/ui/generatorDisplay.ts](../../scripts/ui/generatorDisplay.ts) runs a `system.runInterval` raycasting each player's target block and writing an action-bar line (custom-font glyph + name + amount/cap + progress%). Glyphs live in `resource_packs/idleoregen/font/glyph_E2.png`; each definition's `glyph` is a `\uE2xx` codepoint.

## Documented limitations (v1.0.0)

Carried verbatim from the store page — these define the backlog:

- Not hopper / redstone / piston / comparator compatible.
- Manual collection only; no speed upgrades or boosts; no fuel/catalyst.
- Dragon Egg consumed by the Netherite recipe and not recoverable.
- Netherite generators effectively limited to 2/world in vanilla (Dragon Egg scarcity).

## Known technical risks to carry forward

1. **Single-blob placed state.** `placed_state` is re-read and re-serialized on *every* access and is bound by the per-dynamic-property size limit. It grows with generator count and with any per-instance fields added later (fuel, upgrades). This is the first thing that breaks at scale — addressed in [v1.1.0](v1.1.0-automation-and-recovery.md#foundation-storage-sharding).
2. **Proximity-based entity cleanup** (`getEntities` by radius on break) can mis-target if generators are placed adjacent; tag matching mitigates but relies on the visual entity still existing.
3. **Catalog rewritten every world load** (`initDatabaseAutoUpdate`), which is fine now but means live balance changes ship silently on update — intended, but worth noting for migrations.
