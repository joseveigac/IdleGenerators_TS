# v1.2.0 ŌĆö Progression

Adds permanent per-generator progression and makes generators safe on shared worlds. Builds on v1.1.0's sharded storage, and **introduces the item-in-hand interaction pattern** (held item ŌåÆ apply; empty hand + sneak ŌåÆ toggle; otherwise ŌåÆ withdraw).

---

## Feature 1 ŌĆö Speed / Capacity Upgrades

**Goal.** Let players invest to make a specific generator permanently faster or higher-capacity. Directly answers the store's "No speed upgrades or boosts" limitation. An upgrade is **permanent and per-instance** (distinct from any future temporary boost mechanic).

**Design.**
- Two upgrade axes, each with a few discrete levels:
  - **Speed upgrade** ŌĆö multiplies effective production rate (reduces interval) by a fixed factor per level, e.g. L1 ├Ś1.25, L2 ├Ś1.5, L3 ├Ś2.0.
  - **Capacity upgrade** ŌĆö raises `cap` by a fixed factor or flat amount per level.
- Applied via **held-item + interact** (the addon's item-in-hand interaction convention): holding an `idleoregen:upgrade_speed` / `idleoregen:upgrade_capacity` item and interacting consumes it and increments that instance's level (up to a max). Interaction precedence in `onPlayerInteract`: upgrade item ŌåÆ ownership toggle (empty hand + sneak) ŌåÆ withdraw.
- Effective values are derived, never mutated on the catalog: `effectiveInterval = baseInterval / speedFactor(level)`, `effectiveCap = baseCap * capFactor(level)`. The catalog (`GENERATORS`) stays the immutable balance source; per-instance levels layer on top ŌĆö and any future boost mechanic would apply on top of the upgraded interval the same way.

**Interaction with the lazy model.** Speed level changes `effectiveInterval` going forward. To avoid retroactively re-timing already-accrued production, **settle the buffer at the moment of upgrade**: compute produced-so-far with the old interval, fold it into `storedAmount`, set `lastInteraction = now`, then apply the new level. This mirrors the whole-cycle accounting discipline already in v1.0.0.

**Data model.** `GeneratorData += speedLevel?: number` (default 0) `+ capacityLevel?: number` (default 0). Optional, migration-safe. Bump `DB_VERSION` to 3 with a no-op forward migration (defaults fill in).

**API approach.** Main-hand read via `equippable` (as v1.1.0); no new surfaces. HUD (`generatorDisplay.ts`) and the shared production helper must use the effective interval/cap.

**Balance guardrails.** Cap total speed so a fully-upgraded generator never dwarfs vanilla farming for the same tier. Upgrades should be a meaningful mid/late-game sink, craftable from tier-appropriate materials (e.g. capacity from shulker/copper, speed from redstone/amethyst).

## Feature 2 ŌĆö Multiplayer Ownership & Protection

**Goal.** On servers, stop other players from draining generators you placed. The data is already there: `PlacedInstance.ownerId` is set at placement ([placed.ts](../../scripts/instances/placed.ts) `createPlaced`) but unused by gameplay.

**Design.**
- Each generator has an **access mode**: `private` (owner only) or `public` (anyone can withdraw). Default configurable; recommend `private` for a servers-first audience, `public` if the addon is mostly used in single-player/co-op.
- **Withdraw guard:** in `onPlayerInteract`, before withdrawing, if mode is `private` and `player.id !== ownerId`, deny (silent, or a brief action-bar note) and skip. Upgrade application follows the same guard ŌĆö only the owner modifies their generator.
- **Toggle:** owner sneaks + interacts **with an empty hand** to cycle the access mode (empty-hand disambiguates from withdraw, which gives items, and from upgrade application, which needs a held item). Confirm the new mode via action bar.
- **Break protection (optional, mode-gated):** optionally prevent non-owners from breaking a private generator. Bedrock can't universally cancel block breaking for custom blocks reliably in all contexts, so treat this as best-effort: if `onPlayerBreak` fires for a non-owner on a private generator, re-place/refund is fragile ŌĆö prefer instead to simply not drop the recovery/contents to a non-owner, or leave break open and document it. Decide during implementation; do not over-promise break protection.

**Data model.** `GeneratorData += access?: "public" | "private"` (default per config). `ownerId` already on `PlacedInstance`. Migration-safe default. Bump `DB_VERSION` to 3 (share with upgrades if shipped together).

**Edge cases.**
- Owner leaves the server / world transfers: consider an admin override (ties into v1.3.0's `/idleoregen` command ŌĆö e.g. `reassign`/`claim`).
- `ownerId` missing on very old instances (placed before ownership mattered): treat as `public` or claimable.

**API approach.** `player.id` vs stored `ownerId`; empty-hand check via `equippable` mainhand being undefined/air. No new surfaces.
