# v1.1.0 — Recovery & Expansion

Scope for the next release, kept deliberately small and low-risk: a storage refactor foundation, Dragon Egg recovery, and the first generator **Set**.

Ship order: **Foundation → Dragon Egg Recovery → More Types** (cheapest / lowest-risk first).

**Explicitly not in this version** (moved to [backlog.md](backlog.md)): Advanced (hopper-compatible) Generators, and a resource-boost mechanic — the previously-designed Catalyst System was **dropped**; a different mechanic will be designed later. Redstone Control remains a permanent non-goal.

## Foundation: Storage sharding

**Problem.** `placed_state` is one JSON blob holding every generator ([instances/placed.ts](../../scripts/instances/placed.ts)). It is re-serialized on every CRUD call and bound by the per-dynamic-property byte limit. It grows with generator count today (more Sets = more placements), and later versions add per-instance fields (upgrades, ownership in v1.2.0). Fix it before either axis bites.

**Design.** Shard the placed map into buckets keyed by a coarse spatial hash, keeping each bucket small and only touching the bucket a given position belongs to.

- Bucket key: `sb:w:d:placed:<dim>:<bx>,<bz>` where `bx = floor(x / 512)`, `bz = floor(z / 512)` (region-sized buckets; tune to keep each blob well under the property limit).
- A lightweight index property `sb:w:i:placed_buckets` lists non-empty bucket keys for iteration (offline report, admin queries) without scanning all coordinates.
- `getPlacedAtPos` / `upsertPlaced` / `removePlacedAtPos` keep their signatures; only the internal load/save resolves the right bucket. Callers are unchanged.

**Migration.** `DB_VERSION` 1 → 2. On load, if the legacy `placed_state` exists: read it, re-distribute entries into buckets, write the index, delete the legacy key. Idempotent and one-shot.

**API approach.** `world.getDynamicProperty` / `setDynamicProperty` per bucket; `posToKey` already yields the coordinates needed to compute the bucket. No new APIs.

**Tasks**
- [ ] Bucket resolver + index read/write helpers in `instances/placed.ts` (internal only).
- [ ] Migration path in `storage.ts` gated on `DB_VERSION`.
- [ ] Bump `DB_VERSION` to 2; keep `initDatabaseAutoUpdate` catalog refresh as-is.
- [ ] Verify a large existing world migrates without data loss (in-game).

> **Deferral option.** If v1.1.0 scope is tight, sharding can be reduced to "split into N hash buckets" without the spatial index. The spatial scheme is preferred because v1.3.0's offline report and admin command both benefit from cheap iteration.

## Feature 1 — Dragon Egg Recovery

**Goal.** Get the Dragon Egg back from a Netherite Generator.

**Design (simplified — no script).** The block currently has **no `minecraft:loot` component**, so it drops itself by default ([netherite_generator.block.json](../../behavior_packs/idleoregen/blocks/netherite_generator.block.json)). Add a `minecraft:loot` component pointing to a block loot table that yields `minecraft:dragon_egg` **instead of** the generator. Breaking the Netherite Generator then returns the egg. No changes to `onPlayerBreak` — it already handles instance/entity cleanup; the block's loot handles the drop, and it composes with vanilla break/silk-touch mechanics.

**Trade-off to confirm.** "Drop egg **instead of** itself" means reclaiming the egg **consumes** the generator (you don't get the block back). That's a clean "sacrifice to recover" rule and matches the current framing. If you'd rather keep the generator **too**, the same loot table can list both pools (generator + egg) — a one-line change either way.

**API / files.** Add `"minecraft:loot": "loot_tables/blocks/netherite_generator.json"` to the netherite block; create that loot table dropping `minecraft:dragon_egg` (count 1). Purely declarative.

**Data model.** None. No `PlacedInstance` change, no migration.

## Feature 2 — More Generator Types (organized as Sets)

**Goal.** Grow the catalog as **Sets**, indexed by **Dimension × Category**, so expansion is themed and predictable instead of ad-hoc. Purely additive — the component already handles any type via data.

**Set taxonomy** (the framework going forward):
- **Overworld**
  - *Forests / Wood* ← **this version**
  - *Mobs* → future (home of the Experience Generator, see note)
  - *Ores* (mostly shipped in v1.0.0)
- **Nether** → future (Nether Gold, Glowstone, Nether Wart, Blaze Powder; Quartz already exists)
- **The End** → future (Chorus, Ender Pearl…)

**This version — Overworld / Forests set.** Add a generator for **every wood type**, mirroring the existing `oak_log`: spruce, birch, jungle, acacia, dark oak, mangrove, cherry. (Crimson/warped stems are Nether — defer to the Nether set.) Build-heavy players get passive lumber; it's a natural, low-risk first Set.

**Experience Generator → future "Overworld / Mobs" set.** XP isn't an `ItemStack`, so it can't use the buffer/withdraw path directly — it belongs with a **Mobs set** (mob-drop generators: rotten flesh, bone, string, gunpowder…), where an XP generator would withdraw via `player.addExperience(n)` instead of adding an item. Deferred to that Set; **not** in this version.

**Design notes.**
- Each new type = one `GENERATORS` entry + `blocks/*.block.json` + resource-pack `entity/*.entity.json` + model/texture + `recipes/crafting/*.json` + `.lang` (en_US & es_ES) + a `\uE2xx` glyph. All must stay in sync — a missing file fails silently in-game (see [CLAUDE.md](../../CLAUDE.md)).
- The wood Set can share one `geometry.generator` model with a per-wood texture, keeping authoring cheap.

**Data model.** New catalog entries only. No `PlacedInstance` change.
