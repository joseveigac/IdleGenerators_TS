# v1.3.0 — Admin & Polish

Operator tooling and the feedback/polish that makes the systems built in v1.1–v1.2 feel good. Lowest technical risk of the three phases; highest "feel" return.

---

## Feature 1 — Admin command `/idleoregen`

**Goal.** Give server operators and modpack integrators control without editing code — the store copy explicitly courts modpack compatibility and feedback. Custom slash commands are stable in the target engine (1.21.130+).

**Design — subcommands** (permission-gated to operators where destructive):
- `/idleoregen give <type> [count] [player]` — grant a generator block. Speeds up testing and lets map-makers hand them out.
- `/idleoregen upgrade <speed|capacity> [count] [player]` — grant v1.2 upgrade items.
- `/idleoregen config <interval|cap> <type> <value>` — live-tune balance for a modpack. Writes into the catalog copy (`sb:w:c:generators`); note this currently gets rewritten from code on world load (`initDatabaseAutoUpdate`), so either (a) persist admin overrides in a separate `overrides` property merged on load, or (b) gate the auto-rewrite behind a "no overrides present" check. **Recommend option (a):** a `sb:w:c:generator_overrides` property layered over the code defaults, so updates don't clobber operator tuning.
- `/idleoregen claim` / `reassign <player>` — ownership admin (pairs with v1.2.0 protection): claim ownerless generators, or transfer when a player leaves.
- `/idleoregen stats [player]` — count generators, total buffered, per-type breakdown (uses the sharded index for cheap iteration).

**API approach.** Register in `system.beforeEvents.startup` via the custom-command registry with `CommandPermissionLevel` (operator for destructive subcommands, any for read-only like `stats`). Namespaced `idleoregen:`. Mind the `CustomCommandResult` return-shape quirks (see the `bedrock-addon-dev` custom-commands reference). Enumerated args (generator type / upgrade kind) should use enum arg types so players get autocomplete.

**Data model.** Optional `sb:w:c:generator_overrides` catalog-override property. No `PlacedInstance` change.

**Tasks**
- [ ] Command registration + subcommand routing + enums for type/tier.
- [ ] Overrides property + merge-on-load; adjust `initDatabaseAutoUpdate` to not clobber overrides.
- [ ] Permission gating audit (destructive = operator).
- [ ] `stats` iterates via the sharded bucket index.

## Feature 2 — Sound & particle feedback

**Goal.** Make withdrawing and producing tactile. Currently withdrawal is silent (the `sendMessage`/feedback lines are commented out in [generator.ts](../../scripts/components/generator.ts)).

**Design.**
- **On withdraw:** a short pickup/collect sound + a small particle burst at the block. Distinct subtle cue when the buffer was empty (nothing collected) vs. a successful grab.
- **On upgrade apply (and future held-item actions):** a stronger confirming sound + particle so the (GUI-less) interaction has clear feedback — important since there's no form to confirm the action.
- **Optional ambient:** a very low-frequency, low-volume "working" particle on active generators (throttled hard; skip when many are loaded, or only for the generator the player looks at). Keep strictly optional and performance-bounded.

**API approach.** `dimension.playSound(soundId, location)` and `dimension.spawnParticle(particleId, location)` (or `world.playSound`). Reuse vanilla sound/particle ids first; add custom entries in `sound_definitions.json` only if needed. Drive feedback from the existing interaction handlers — no new tick loops except the (optional, throttled) ambient.

**Data model.** None.

**Tasks**
- [ ] Choose vanilla (or define custom) sound + particle ids for withdraw / apply.
- [ ] Wire into interact/upgrade branches.
- [ ] Guard any ambient effect behind a strict throttle + loaded-count check.

## Feature 3 — Offline-earnings report

**Goal.** Reward re-engagement: when a player returns, tell them what their generators made while they were away — the classic idle-game dopamine loop.

**Design.**
- On player join, compute the total **claimable** across generators the player owns (uses `ownerId` from v1.2.0 + the sharded index from v1.1.0 for cheap iteration).
- Show a concise summary via action bar or chat: e.g. `Welcome back! Your 6 generators are holding 1,204 items ready to collect.` Optionally a per-type top-3 line.
- **Honesty about the cap:** buffers are capped, so this is "currently claimable," not "produced since you left." Phrase it as *ready to collect* to stay accurate rather than implying uncapped accrual. (A true "since last seen" number would need a per-player `lastSeen` stamp and per-generator production history — heavier; defer unless players ask.)

**API approach.** `world.afterEvents.playerSpawn` (check the `initialSpawn` flag) → iterate the player's owned instances via the bucket index → sum derived `stored` (using the shared effective-value helper so upgraded buffers read correctly) → `player.sendMessage` / action bar.

**Data model.** None required for the "claimable" version. A future "since last seen" version would add `lastSeen` to player data (the player-DP helpers are already stubbed in [storage.ts](../../scripts/storage/storage.ts)).

**Tasks**
- [ ] `playerSpawn` handler gated on `initialSpawn`.
- [ ] Owned-instance iteration via bucket index; sum via shared production helper.
- [ ] Concise, accurate summary message (en_US + es_ES).
